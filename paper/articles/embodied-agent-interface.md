# Embodied Agent Interface Blog

## 2.9-2.23周报.md

在知识层面，我最大的收获是把决策—控制—物理这条链条理清楚了一些。具身任务表面上是语言到动作，但中间至少要经过目标形式化、子目标组织、动作可执行性检查，然后才是控制器如何在真实动力学和接触约束下把动作落地。博客里涉及的传统控制/动力学内容，很多时候sim2real失败并不是策略不够强，而是给策略的动作接口、控制频率、延迟、接触模型、传感器噪声已经决定了它能否稳定地把意图变成状态转移。换到我自己的场景里，这一点直接对应 go2-x5 的 sim2sim/sim2real：策略在一个仿真器里看起来合理，但换到另一个仿真器或真实系统，控制链路与动力学假设变了，策略就可能出现同样的动作、不同的结果。

同时，这篇博客也把评测这件事讲得更有操作性。以前我更习惯用 success rate 去总结一个系统，但 success rate 很难指导下一步改什么。EAI 这类思路更强调把任务过程拆成可诊断的中间步骤：目标理解有没有偏、子目标是不是合理、动作序列是否可执行、状态转移假设是否成立。这样评测不只是给分，而是把失败定位到具体环节，能形成直接的改进路径。对我而言，这种诊断式评测和传统控制的那套问题归因是相通的：先把系统拆开，再找最先失效的那一段，而不是在终点指标上反复调参。

我自己的思考是，后续如果要做评测/benchmark 或者推进工程，不应该先从任务数量、模型大小这些表面变量入手，而应该先把链条里的约束说清楚：动作接口是什么、控制频率是多少、状态估计给到什么、接触与动力学偏差主要在哪里；在这些约束下再去定义中间产物与错误类型，才能保证评测结果有可解释性，也能保证工程改动有归因。否则就很容易出现一种情况：上层看起来决策没问题，但下层控制与动力学假设一变，整个系统表现就被冲掉，最后只能用更大模型或更多数据去硬扛。
